# 🤖 AI 开发工作流规范 (Dev Only v1.6.0 / local)

本文档规定了本项目内 AI 助手的默认工作模式。当前模式为 `local`。

---

## 0. 会话启动规则

- 每次开启新会话时，AI 默认进入 **Dev 模式**。
- 标准开场回复格式：

```text
✅ 已进入开发模式
📄 正在加载规范：AGENTS.md
📋 开始执行工作流 Step 1...
```

- 若用户在消息开头使用 `!quick`，AI 进入轻量流程，仅适用于小范围、低风险修改。**禁止**将 `!quick` 用于架构重构、接口调整、数据模型变化或依赖外部上下文的大改动。

---

## 1. 核心指令

1. **本地上下文优先原则**：执行任何任务前，应先读取 `docs/` 下可用的 API 文档、设计文档、测试资料和其他实现细节文档；若无本地文档，再回退到 `SPEC.md`、源码和工程配置。
2. **上下文来源优先级**：
   - `docs/` 下的 API 文档、设计文档和其他实现细节文档（若存在）
   - `SPEC.md`
   - 源码、工程配置、`CHANGELOG.md`、`devlogs/` 最新 5 份日志
3. **禁止臆造外部契约**：若需求文档、接口文档或测试资料缺失，必须明确指出缺口，不得伪造字段、流程、接口约定或验收口径。
4. **SPEC 定位原则**：`SPEC.md` 只用于记录项目整体结构、项目定位、技术栈、编程语言、框架选型、模块边界和全局约束，不承载 API 字段、接口细节、详细设计方案等细粒度文档。
5. **文档归档原则**：API 文档、设计文档、接口约定、数据结构细节、流程细节等内容应放在 `docs/` 目录维护。
6. **SPEC 首次初始化原则**：若 `SPEC.md` 不存在或仍为空模板，必须先执行下文的「SPEC 自动初始化流程」，再开始编码。
7. **日志记录原则**：所有实质开发活动都必须在 `devlogs/` 下记录。开始工作前先建草稿，完成后回填状态、验证结果与变更文件。
8. **本地文档缺失处理原则**：若 `docs/` 缺失或无相关文档，不阻塞开发流程；AI 可继续基于 `SPEC.md`、源码、工程配置和用户明确指令执行任务，但不得臆造需求细节、接口契约、测试标准或验收口径，无法确认部分必须标记为“待确认”。
9. **项目规范识别原则**：初始化或编码前，必须读取项目内 `skills/project-standards/SKILL.md`，优先识别用户自定义规范文件和项目现有 lint / format / build / test 配置；识别出的项目规范优先于语言默认惯例。
10. **项目规范缺失处理原则**：若未识别到明确项目规范，可做保守推断，但必须标记为“待确认”，不得把推断结果当成项目既有规范。
11. **Code Review 触发原则**：当任务涉及接口变更、数据模型变更、数据库迁移、跨模块修改、服务端核心逻辑、鉴权、中间件、异步任务或高风险重构时，完成实现后必须执行一次 Code Review，按项目内 `skills/code-review/SKILL.md` 规范执行。
12. **Code Review 结论原则**：Code Review 必须输出问题清单、风险点、建议处理项和结论（`pass` / `pass with risk` / `blocked`）。测试通过不等于 Review 通过。

## 2. SPEC.md 定位与模板规范

当 AI 需要创建或重构 `SPEC.md` 时，必须保持在“项目级基础规范”范围内，严格包含以下章节：

1. **项目定位与背景**：项目目标、使用场景、当前边界。
2. **技术栈与运行环境**：语言、框架、关键依赖、运行方式。
3. **项目结构与模块边界**：目录结构、模块职责、服务边界。
4. **全局约束与编码约定**：通用规范、跨模块约束、基础约定。
5. **集成与依赖概览**：外部系统、数据库、中间件、核心依赖概况。
6. **非功能性需求**：性能、安全性、稳定性、兼容性。
7. **变更记录**：版本历史与关联日志。

详细 API 文档、详细设计文档、接口字段定义、流程说明等内容应单独维护在 `docs/` 目录。

## 3. 开发日志规范 (devlogs)

文件名格式：`YYYY-MM-DD-dev-任务名.md`

每个日志文件必须以以下 YAML frontmatter 开头：

```yaml
---
date: YYYY-MM-DD
role: dev
task: 任务简述
related_task_id:
files_changed:
  - src/xxx.py
  - SPEC.md
status: draft | in_progress | done | blocked
---
```

日志要求：

1. 开始实质开发前创建日志草稿。
2. 完成后回填 `files_changed`、验证结果和最终 `status`。
3. 若交付内容已具备验证条件，在日志末尾注明 `状态: 可测试 ✅`。
4. 获取历史上下文时，最多读取 `devlogs/` 目录下最新 5 份日志。

## 4. 标准工作流

### Step 0 工程状态与 SPEC 初始化检测

- 若找不到 `src/`、`app/`、`server/`、`cmd/`、`package.json`、`pyproject.toml`、`requirements.txt`、`go.mod`、`pom.xml` 等工程入口或配置文件，说明代码工程尚未初始化。
  - **行为**：主动询问用户计划使用的技术栈，例如 `Python`、`Node.js`、`React + Vite` 等，再协助生成基础骨架。
- 若 `SPEC.md` 不存在或仍为空模板，必须执行下方的「SPEC 自动初始化流程」。

### SPEC 自动初始化流程

1. 列出项目根目录文件，识别工程配置、源码目录、服务目录、测试目录、`docs/` 目录和关键文档。
2. 若存在代码目录或配置文件，读取入口文件、目录树、依赖信息，以及关键服务端资产，例如控制器、服务层、模型、数据库迁移、消息队列任务、中间件或定时任务；同时按 `skills/project-standards/SKILL.md` 扫描项目现有 lint / format / build / test 规范和用户自定义规范文件。
3. 若 `docs/` 下存在相关 API、设计或测试资料，优先读取；若 `docs/` 缺失或为空，则继续后续步骤，并将缺失资料标记为“待确认”。
4. 同步阅读 `SPEC.md`、`CHANGELOG.md` 和 `devlogs/` 最新 5 份日志。
5. 严格按本规范第 2 节的 7 个章节生成 `SPEC.md`。
6. 所有内容必须基于扫描到的真实信息和用户明确指令；无法确认的部分必须明确写为“待确认”或“暂无”。

### Step 1 读取上下文

1. 优先读取 `docs/` 下相关 API 文档、设计文档和测试资料（若存在）。
2. 同步阅读：
   - `SPEC.md`
   - 源码和工程配置
   - `CHANGELOG.md`
   - `devlogs/` 最新 5 份日志
3. 若 `docs/` 缺失或无相关文档，不阻塞；继续基于 `SPEC.md`、源码、工程配置和用户明确指令执行。
4. 对需求细节、接口契约、测试标准无法确认的部分，必须明确标记为“待确认”。

### Step 2 创建或更新 Dev 日志

在 `devlogs/` 中创建或更新本次任务日志草稿，记录任务背景、计划修改范围、任务拆分草案和待验证项。

### Step 3 核对 SPEC

编码前先读取 `skills/project-standards/SKILL.md`，识别项目已有规范配置和用户自定义规范文件，并确认 `SPEC.md` 的“全局约束与编码约定”章节是否覆盖这些内容。若未识别到明确规范，只能保守推断并标记为“待确认”。若缺的是详细 API、详细设计或数据结构细节，应补充到 `docs/` 目录；若缺的是项目级基础约束，再更新 `SPEC.md`。

### Step 4 列出任务清单

基于本地文档、当前代码和 `SPEC.md`，先输出本次需求的任务清单，至少包含：

- 任务拆分
- 影响范围
- 风险点
- 验证项
- 预计修改文件

在这一步**不得直接开始编码**。

### Step 5 等待确认

将任务清单提交给用户确认。用户确认前，不得开始实施；若用户调整范围或优先级，必须先更新任务清单和日志，再继续后续步骤。

### Step 6 编写代码

根据已确认的上下文和 `SPEC.md` 执行实现。若用户使用 `!quick`，只允许进行局部、低风险修改。

### Step 7 同步技术规范

若代码改动引入了新的项目级结构、模块边界、语言/框架约束或架构决策，必须同步更新 `SPEC.md`；若改动的是 API、接口字段、详细设计或流程细节，必须同步更新 `docs/` 下对应文档。

### Step 8 验证

完成实现后执行必要验证，至少覆盖：

- 受影响模块的最小可运行检查
- 相关测试或样例验证
- 服务端接口、数据持久化、异步任务或中间件链路的受影响验证（若存在）
- 可用测试资料中的相关用例覆盖确认（若存在）

### Step 9 Code Review

当本次改动满足 Code Review 触发条件时，必须读取 `skills/code-review/SKILL.md` 并按其中规则完成审查。

Review 至少应覆盖：

- 逻辑正确性
- 回归风险
- 边界条件与异常处理
- 接口兼容性
- 数据一致性
- 安全性
- 性能与资源影响
- 服务端副作用链路（若存在）

Review 输出至少包含：

- Findings
- 风险点
- 建议处理项
- 结论（`pass` / `pass with risk` / `blocked`）

若结论为 `blocked`，不得进入收尾确认；若结论为 `pass with risk`，必须先向用户明确说明风险后再继续。

### Step 10 收尾确认

回填开发日志的 `files_changed`、验证结果与最终 `status`。若当前交付物已具备测试条件，在日志末尾注明 `状态: 可测试 ✅`。
